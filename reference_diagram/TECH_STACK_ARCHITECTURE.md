# 아키텍처 및 기술 스택 도입 배경 (Tech Stack Justification)

본 문서에서는 '인생두컷' 프로젝트가 단순한 프로토타입을 넘어, 양산 수준의 서비스 안정성과 실시간성을 확보할 수 있도록 설계한 핵심 기술 스택의 도입 배경과 구조적 의도를 기술합니다.

---

## 1. 로봇 제어 및 미들웨어 (Robot & Middleware)

### 1.1. ROS 2 (Humble) & Ubuntu 22.04
* **선택 배경:** 로봇 하드웨어 I/O 및 멀티 프로세스 통신의 코어 프레임워크로 채택했습니다.
* **설계 의도 (Justification):** 프로덕션 환경에서는 시스템의 장기적인 안정성이 최우선입니다. 기능 추가 위주의 Non-LTS 버전(Galactic, Iron 등)에서 발생할 수 있는 갑작스러운 지원 중단(Deprecated) 리스크를 배제하고, 5년간 안정적인 보안 및 패치 업데이트가 보장되는 LTS 버전인 Humble을 도입하여 전체 시스템 런타임의 강건성(Robustness)을 확보했습니다.

### 1.2. Eclipse Cyclone DDS
* **선택 배경:** ROS 2 Humble의 기본 RMW(Robot Middleware)인 Fast DDS 대신, 네트워크 통신 계층을 Cyclone DDS로 전면 교체했습니다.
* **설계 의도 (Justification):** RealSense의 대용량 RGB-D 데이터와 고해상도 비전 토픽을 멀티캐스트 환경에서 전송할 때 발생하는 패킷 드롭 및 노드 디스커버리 지연 현상을 해결해야 했습니다. 높은 대역폭 처리와 노드 간 연결 안정성이 압도적으로 우수한 Cyclone DDS를 적용하여 멀티 노드 통신 병목을 원천 차단했습니다.

## 2. 하드웨어 구성 (Hardware)

### 2.1. Doosan M0609 (협동로봇)
* **선택 배경:** 포토부스 서비스의 특성상 고객과 로봇이 근거리에서 상호작용(HRI)하는 환경을 위한 메인 매니퓰레이터입니다.
* **설계 의도 (Justification):** 일반적인 협동로봇(예: UR 등 모터 전류 기반 감지)과 달리, 두산 M 시리즈는 6개의 모든 조인트에 고해상도 토크 센서가 내장되어 있습니다. 미세한 외력에도 즉각적으로 반응하는 민감한 충돌 감지와 순응 제어(Compliance Control)를 구현함으로써 대고객 서비스에 필수적인 '물리적 안전성'을 극대화했습니다.

### 2.2. OnRobot RG2 (그리퍼)
* **선택 배경:** 다양한 규격의 소품(왕관, 요술봉 등)을 집고 건네주기 위한 전동식 툴입니다.
* **설계 의도 (Justification):** 공압식 그리퍼는 컴프레서 등 부가 인프라가 필요해 포토부스 내장에 부적합합니다. 반면 플러그 앤 플레이 방식의 RG2는 ROS 2 드라이버 호환성이 뛰어나며, 파지력(Force)과 폭(Width)을 소프트웨어적으로 정밀하게 가변 제어할 수 있어 다품종 소품 핸들링에 최적화된 아키텍처를 제공합니다.

## 3. AI 및 비전 프로세싱 (AI & Vision)

### 3.1. YOLOv8 + MediaPipe 혼용 (Hybrid Vision)
* **선택 배경:** 안전 감시, 소품 인식, 고객 제스처 파악을 위한 실시간 비전 시스템입니다.
* **설계 의도 (Justification):** 역할 분담을 통한 컴퓨팅 리소스 최적화가 핵심입니다. 전체 공간의 소품 인식과 안전 구역(Safety Zone) 침범 등 글로벌 컨텍스트는 GPU 기반의 YOLOv8로 처리하여 높은 신뢰도를 확보했습니다. 동시에 고객의 빠르고 미세한 손가락 제스처(따봉 등) 트래킹은 랜드마크 추출에 특화된 초경량 프레임워크 MediaPipe를 병렬 연산으로 분리하여, CPU 자원 낭비를 막고 실시간 제어 루프의 응답성을 극대화했습니다.

### 3.2. OpenAI (Whisper & GPT-4o)
* **선택 배경:** 자연스러운 사용자 음성 명령으로부터 의도(Intent)를 파악하기 위한 자연어 처리 계층입니다.
* **설계 의도 (Justification):** 단순 규칙(Rule-based) STT를 사용하면 발화 텍스트와 소품(Entity)을 1:1로 매핑하는 코드가 하드코딩되어 확장성이 제한됩니다. Whisper로 텍스트를 추출하고 GPT-4o의 구조화된 JSON 반환 기능을 거침으로써, "공주 컨셉으로 해줘" 같은 자연스러운 문맥 속에서 필요한 소품(왕관, 요술봉) 데이터를 프로그래밍 가능한 형태로 완벽히 추출해 냈습니다.

## 4. 백엔드 및 인프라 (Backend & Infra)

### 4.1. Firebase Realtime DB
* **선택 배경:** 키오스크 PC와 로봇 제어 PC 간의 제어 신호 동기화를 위한 Serverless 데이터베이스입니다.
* **설계 의도 (Justification):** RDBMS(MySQL, AWS RDS 등)의 복잡한 트랜잭션 연산이 아닌, 분리된 디바이스 간의 '밀리초 단위 상태 동기화(State Sync)'가 주 목적입니다. 웹소켓(WebSocket) 서버를 직접 구축하거나 MQTT 브로커를 띄우는 인프라 관리 비용을 배제하고, 클라이언트리스 환경에서 양방향 데이터 바인딩을 보장하는 Firebase를 채택하여 이벤트 기반 아키텍처의 네트워크 오버헤드를 최소화했습니다.

### 4.2. Docker & Docker Compose
* **선택 배경:** ROS 2 코어 시스템과 AI 비전 시스템을 분리하기 위한 컨테이너 환경입니다.
* **설계 의도 (Justification):** 호스트 PC에 YOLOv8과 최신 딥러닝 패키지(예: Numpy 2.x)를 직접 설치하면, ROS 2 Humble 시스템 패키지(Numpy 1.x 요구)와 치명적인 의존성 충돌이 발생합니다. 하드웨어 제어 노드는 실시간성을 위해 호스트 OS(Ubuntu)에 네이티브로 남기고, 무거운 딥러닝 종속성을 지닌 AI 노드만 도커로 완벽히 격리(Isolation)하는 하이브리드 아키텍처를 설계해 시스템 크래시를 원천 방지했습니다.

---

## 5. 프론트엔드 (Frontend)

### 5.1. HTML / CSS / Vanilla JS
* **선택 배경:** 키오스크 모니터와 사용자 인터랙션을 담당하는 임베디드 웹 어플리케이션입니다.
* **설계 의도 (Justification):** 범용 SPA(Single Page Application)와 달리 이 프로젝트의 웹앱은 단일 스크린에서 영상 스트림 수신과 UI 애니메이션을 보여주는 임베디드 단일 목적(Single Purpose)성을 띱니다. React나 Vue의 가상 DOM 생성 및 복잡한 상태 관리(Redux 등) 과정에서 발생하는 런타임 오버헤드를 제거했습니다. 제한된 하드웨어 자원에서 DOM에 직접 접근해 딜레이 없이 가장 빠르고 가볍게 렌더링하기 위해, 오버엔지니어링을 배제하고 Vanilla JS를 채택했습니다.
